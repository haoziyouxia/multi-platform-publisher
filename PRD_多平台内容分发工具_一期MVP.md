## 二、项目背景

### 2.1 业务背景

个人创作者在日常运营中面临一个高频痛点：一篇内容写好之后，需要分别登录小红书、公众号后台、头条号创作者平台，手动进行排版、上传图片、填写标题和正文，然后逐个发布。三个平台的操作习惯和格式要求不同，每次发布耗时 10-15 分钟，且容易出错。

### 2.2 核心问题

| 问题 | 描述 | 严重程度 |
|------|------|:--:|
| 多平台切换 | 每发一篇需登录 3 个后台，反复切换 | 🔴 高 |
| 格式不兼容 | 图片尺寸、排版规则不同，需手动调整 | 🔴 高 |
| 无统一管理 | 发布历史分散，无法追踪各平台状态 | 🟡 中 |
| 效率低下 | 人工逐平台发布平均耗时 12.3 分钟/篇 | 🔴 高 |

### 2.3 现有方案分析

- **手动发布**: 当前主流方式，耗时且易出错
- **竞品工具**: 新榜小豆芽、聚媒通等覆盖平台广，但偏向企业/MCN，个人创作者使用门槛和成本较高
- **技术现状**: 小红书和今日头条无官方图文发布 API，公众号个人订阅号 API 权限受限——三个平台均需采用浏览器自动化方案

### 2.4 一期目标

> 为个人创作者提供一个轻量 Web 管理后台，**一次编辑、三平台发布**，将单篇内容的多平台发布耗时从 12+ 分钟降至 3 分钟以内。

## 四、功能清单与需求描述

> 以下需求采用 **EARS 原则**编写，确保每条需求清晰、可验证、无歧义。

### 4.1 模块一：账号管理

#### 4.1.1 账号绑定

| 编号 | 类型 | 需求描述 |
|:--:|------|------|
| FR-01 | Ubiquitous | The system shall 支持用户绑定小红书、微信公众号、今日头条三个平台的账号，每个平台至少绑定 1 个、最多绑定 3 个账号 |
| FR-02 | Event-driven | When 用户点击"绑定账号"按钮，the system shall 展示对应的平台登录页面（或二维码），引导用户完成授权 |
| FR-03 | State-driven | While 用户尚未绑定任一平台账号，the system shall 在首页展示引导提示，并禁用内容发布功能 |
| FR-04 | Unwanted | If 平台授权超时（超过 5 分钟未扫码/登录），then the system shall 提示用户"授权已超时，请重新绑定"并终止当前绑定流程 |
| FR-05 | Unwanted | If 账号绑定失败（如密码错误、风控拦截），then the system shall 展示失败原因并允许用户重试 |

#### 4.1.2 登录态维护

| 编号 | 类型 | 需求描述 |
|:--:|------|------|
| FR-06 | Ubiquitous | The system shall 持久化保存已绑定账号的浏览器会话（Cookie/Session），避免每次发布时重新登录 |
| FR-07 | State-driven | While 检测到某平台登录态已过期，the system shall 在该平台对应的发布入口展示"需重新授权"标识 |
| FR-08 | Event-driven | When 用户点击"重新授权"，the system shall 引导用户完成重新登录，并更新本地保存的会话状态 |

#### 4.1.3 账号解绑

| 编号 | 类型 | 需求描述 |
|:--:|------|------|
| FR-09 | Event-driven | When 用户确认解绑某平台账号，the system shall 清除该账号的会话数据并更新绑定状态 |

---

### 4.2 模块二：内容编辑

#### 4.2.1 统一编辑器

| 编号 | 类型 | 需求描述 |
|:--:|------|------|
| FR-10 | Ubiquitous | The system shall 提供一个统一的富文本编辑器，支持用户输入标题和正文内容 |
| FR-11 | Ubiquitous | The system shall 支持用户在正文中上传本地图片（支持 JPG/PNG/GIF 格式，单张 ≤ 10MB） |
| FR-12 | Ubiquitous | The system shall 实时保存编辑器内容为草稿（本地存储），防止用户误关闭导致内容丢失 |
| FR-13 | Event-driven | When 用户在编辑器中粘贴来自其他来源（如 Word、网页）的内容，the system shall 清除冗余样式并保留基本格式（加粗、斜体、列表、链接） |
| FR-14 | Optional | Where 用户上传了多张图片，the system shall 支持拖拽调整图片顺序 |

#### 4.2.2 平台差异化编辑（P1）

| 编号 | 类型 | 需求描述 |
|:--:|------|------|
| FR-15 | Event-driven | When 用户选择"多平台发布"，the system shall 默认将统一编辑器的内容同步到所有已选平台 |
| FR-16 | Event-driven | When 用户切换到某个平台的"独立编辑"模式，the system shall 允许用户对该平台的标题/正文做微调，不影响其他平台的内容 |
| FR-17 | State-driven | While 某平台处于"独立编辑"模式，the system shall 在平台标签上展示"已自定义"标识，提示用户该平台内容与统一内容不同 |

#### 4.2.3 发布前预览（P1）

| 编号 | 类型 | 需求描述 |
|:--:|------|------|
| FR-18 | Event-driven | When 用户点击"预览"按钮，the system shall 展示内容在各平台客户端的模拟效果（包括标题截断、图片比例、排版差异） |
| FR-19 | Optional | Where 预览显示图片可能被平台裁剪，the system shall 标注可能的裁剪区域并提示用户调整 |

---

### 4.3 模块三：内容发布

#### 4.3.1 发布执行

| 编号 | 类型 | 需求描述 |
|:--:|------|------|
| FR-20 | Event-driven | When 用户选择目标平台并点击"发布"，the system shall 并行向所选平台提交发布任务 |
| FR-21 | Ubiquitous | The system shall 在发布过程中展示实时进度（如"正在发布到小红书…"、"公众号发布完成"） |
| FR-22 | Event-driven | When 所有平台发布完成（成功或失败），the system shall 汇总展示发布结果 |
| FR-23 | Unwanted | If 任一平台发布失败，then the system shall 展示具体失败原因（如"审核不通过"、"图片格式不支持"、"登录态过期"），且不影响其他平台的发布结果 |
| FR-24 | Unwanted | If 发布任务提交后平台审核不通过，then the system shall 在发布历史中更新状态为"审核未通过"，并展示平台返回的驳回原因 |
| FR-25 | Event-driven | When 用户点击"重试发布"，the system shall 仅对上次失败的平台重新提交发布任务 |

#### 4.3.2 图片预处理

| 编号 | 类型 | 需求描述 |
|:--:|------|------|
| FR-26 | Ubiquitous | The system shall 在发布前校验图片是否符合目标平台要求，不满足时给出明确提示 |

各平台图片要求如下：

| 平台 | 推荐比例 | 推荐尺寸 | 最大文件 | 其他 |
|------|:--:|------|------|------|
| 小红书 | 3:4（竖版） | 1242×1660px | 20MB | 全篇统一比例 |
| 公众号 | 2.35:1（封面） | 900×383px（封面） | 10MB | 正文图片宽度建议 ≤ 1080px |
| 今日头条 | 16:9 或 4:3 | ≥ 600×400px | — | 封面自动抓取正文首图 |

> **说明**: 一期 MVP 仅做校验+提示，不做自动裁剪（自动裁剪列入 P2）

#### 4.3.3 定时发布（P2）

| 编号 | 类型 | 需求描述 |
|:--:|------|------|
| FR-27 | Optional | Where 用户设置了定时发布时间，the system shall 在指定时间自动触发发布任务 |
| FR-28 | Unwanted | If 定时发布时发现登录态已过期，then the system shall 取消本次定时发布并通知用户 |

---

### 4.4 模块四：发布管理（P1）

#### 4.4.1 发布历史

| 编号 | 类型 | 需求描述 |
|:--:|------|------|
| FR-29 | Ubiquitous | The system shall 以列表形式展示用户的发布历史，每条记录包含：标题、发布平台、发布时间、当前状态 |
| FR-30 | Ubiquitous | The system shall 支持按平台、状态、时间范围筛选发布历史 |

#### 4.4.2 发布状态

| 编号 | 类型 | 需求描述 |
|:--:|------|------|
| FR-31 | Ubiquitous | The system shall 维护每条发布记录的以下状态之一：发布中 → 已发布 / 审核中 / 审核未通过 / 发布失败 |
| FR-32 | State-driven | While 内容处于"发布中"状态，the system shall 展示加载动画，并每 30 秒轮询一次结果直至状态确定 |
| FR-33 | Event-driven | When 发布结果确定后，the system shall 在发布历史中更新对应记录的状态，并通过页面通知告知用户 |

### 5.3 异常流程矩阵

| 阶段 | 异常场景 | 系统行为 | 用户感知 |
|------|------|------|------|
| 账号绑定 | 二维码过期 | 终止绑定流程 | "授权已超时，请重新绑定" |
| 账号绑定 | 登录态过期 | 标记平台状态 | 平台标签展示"需重新授权" |
| 内容编辑 | 浏览器意外关闭 | 恢复本地草稿 | 重新打开后提示"是否恢复上次编辑内容" |
| 图片上传 | 图片过大/格式不支持 | 拒绝上传并提示 | "图片大小需 ≤10MB，支持 JPG/PNG/GIF" |
| 发布前 | 图片不符合平台要求 | 阻止发布并提示 | "小红书要求图片为3:4比例，当前图片为1:1，建议调整" |
| 发布中 | 单平台发布失败 | 隔离失败，其他继续 | "公众号发布成功，小红书发布失败：登录态过期" |
| 发布中 | 全平台发布失败 | 展示汇总错误信息 | 逐一展示每个平台失败原因 |
| 发布后 | 平台审核不通过 | 更新状态+通知 | "头条号内容审核未通过：含违规内容" |
| 重试发布 | 连续失败 | 允许重试但建议人工 | "发布已连续失败 2 次，建议手动登录平台检查" |
| 网络 | 发布过程中断网 | 保留发布任务 | 网络恢复后提示"有未完成的发布任务" |

## 七、数据指标

### 7.1 核心指标（一期关注）

| 指标 | 定义 | 目标值 | 数据来源 |
|------|------|:--:|------|
| 绑定完成率 | 完成至少 1 个平台绑定的用户数 / 注册用户数 | ≥ 80% | 后台统计 |
| 发布成功率 | 成功提交发布的任务数 / 总发布任务数 | ≥ 85% | 后台统计 |
| 平均发布耗时 | 从点击发布到所有平台提交完成的时间 | ≤ 3 分钟 | 后台计时 |
| 单篇多平台覆盖率 | 同一内容发布到 ≥2 个平台的比例 | ≥ 60% | 后台统计 |
| 次日留存 | 发布过内容的用户次日仍使用的比例 | ≥ 40% | 埋点 |

### 7.2 埋点需求

| 事件名 | 触发时机 | 属性 |
|------|------|------|
| `account_bind_start` | 用户开始绑定账号 | 平台、方式（扫码/密码） |
| `account_bind_result` | 绑定完成（成功/失败） | 平台、结果、失败原因 |
| `editor_open` | 进入内容编辑器 | 来源（新建/草稿恢复） |
| `editor_image_upload` | 上传图片 | 图片数量、总大小 |
| `publish_click` | 点击发布按钮 | 目标平台数、平台列表 |
| `publish_result` | 发布任务完成 | 各平台结果、耗时 |
| `publish_retry` | 重试发布 | 平台、上次失败原因 |
| `page_view` | 页面浏览 | 页面路径、停留时长 |

## 八、技术约束与依赖

### 8.1 关键技术方案

| 模块 | 方案 | 说明 |
|------|------|------|
| 公众号发布 | Playwright 浏览器自动化 | 个人订阅号无 API 权限，模拟网页端发布 |
| 小红书发布 | Playwright 浏览器自动化 | 无官方图文发布 API，模拟网页端发布 |
| 今日头条发布 | Playwright 浏览器自动化 | 官方 API 仅支持视频，模拟网页端发布 |
| 会话管理 | 浏览器上下文持久化 | 保存 Cookie/Session 避免反复登录 |
| 反检测 | playwright-extra + stealth 插件 | 隐藏自动化特征，降低风控概率 |

### 8.2 依赖与风险

| 依赖/风险 | 影响 | 应对措施 |
|------|------|------|
| 平台前端页面改版 | 自动化脚本可能失效 | 关键选择器可配置化；监控+告警 |
| 平台风控升级 | 发布成功率下降 | 随机延迟+真人模拟；多账号轮换（后续） |
| 浏览器资源消耗 | 并发发布时资源占用高 | 一期限制同时发布任务数 ≤ 3 |
| 公众号个人号群发次数限制 | 每天仅 1 次 | 发布前提示用户确认；超出限制时阻止 |

## 九、验收标准

### 9.1 功能验收

| # | 验收项 | 验收条件 | 优先级 |
|:--:|------|------|:--:|
| AC-01 | 账号绑定 | 用户可通过扫码完成小红书、公众号、头条号的绑定，绑定成功后展示已绑定状态 | P0 |
| AC-02 | 会话复用 | 绑定完成后关闭浏览器再打开，无需重新登录即可直接编辑发布 | P0 |
| AC-03 | 内容编辑 | 支持输入标题、正文，支持加粗、斜体、列表、链接等基本格式 | P0 |
| AC-04 | 图片上传 | 支持上传 JPG/PNG/GIF 图片，单张 ≤ 10MB，至少支持一次上传 9 张图 | P0 |
| AC-05 | 草稿保存 | 关闭编辑器后重新打开，上次编辑内容自动恢复 | P0 |
| AC-06 | 一键发布 | 同时选择 3 个平台点击发布，3 个平台均能成功提交发布任务 | P0 |
| AC-07 | 发布进度 | 发布过程中展示每个平台的实时进度 | P0 |
| AC-08 | 发布结果 | 发布完成后展示各平台的成功/失败状态和原因 | P0 |
| AC-09 | 失败隔离 | 任一平台发布失败不影响其他平台，失败平台可单独重试 | P0 |
| AC-10 | 图片校验 | 图片不满足平台要求时，在发布前给出明确提示 | P0 |
| AC-11 | 平台差异化编辑 | 可对单个平台的标题/正文做独立修改，不影响其他平台 | P1 |
| AC-12 | 发布历史 | 可查看发布历史列表，支持按平台、状态、时间筛选 | P1 |
| AC-13 | 发布状态追踪 | 审核中的内容自动轮询状态，审核不通过时更新状态并通知 | P1 |
| AC-14 | 登录态过期处理 | 平台登录态过期时展示"需重新授权"，用户可一键重新绑定 | P1 |

### 9.2 非功能验收

| # | 验收项 | 验收条件 |
|:--:|------|------|
| NAC-01 | 性能 | 单次发布（3 平台并行）提交耗时 ≤ 3 分钟 |
| NAC-02 | 可用性 | 发布成功率 ≥ 85%（排除平台自身审核驳回） |
| NAC-03 | 浏览器兼容 | 支持 Chrome 90+、Edge 90+ |
| NAC-04 | 安全 | 用户会话数据加密存储，不在前端暴露敏感 Cookie |
